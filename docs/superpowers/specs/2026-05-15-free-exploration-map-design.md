# 自由探索地图扩展设计（轻量挂接型）

## 1. 背景与目标

当前项目已经具备基础探索能力：主线可进入探索阶段，玩家执行一个探索动作后获得奖励，再返回主线。现状主要体现在以下位置：

- 探索数据以单个场景和动作列表建模，见 `src/game/data/exploration.ts:3`
- 探索动作执行后直接切回主线节点，见 `src/game/engine/explorationEngine.ts:46`
- 类型层支持探索奖励、探索动作和探索场景，但尚未支持地图节点、商店、货币和支线状态，见 `src/game/types.ts:34`、`src/game/types.ts:56`

本设计的目标是把现有“一次动作式探索”升级为“轻量挂接型自由探索地图”：

1. 在特定主线节点前进入一张探索地图
2. 地图节点全部可见
3. 玩家可无限次浏览地图节点，不设行动点上限
4. 商店、支线、武学、情报都以固定内容形式预设
5. 玩家必须手动点击“结束探索 / 返回主线”后，才离开探索地图并继续主线

## 2. 范围

### 2.1 首版范围内

- 新增货币：银两
- 新增探索地图层与地图节点层
- 支持五类节点：商铺、支线、武学、情报、离开
- 地图内行为执行后仍停留在探索地图
- 玩家手动结束探索后才返回主线
- 将现有三处探索内容扩成三张小地图：酒楼、渡口、盐港

### 2.2 首版不做

- 行动力 / 步数限制
- 随机刷新商品或支线
- 隐藏地图节点与发现机制
- 地图内连续战斗链
- 多阶段支线任务链
- 复杂经济系统（讨价还价、动态价格、库存刷新）

## 3. 方案选择

本次采用“在现有 exploration phase 上扩一层地图壳”的方案，而不是直接重写为统一世界图。

选择理由：

- 与现有架构兼容性最好，尤其适配当前 `ExplorationScreen` 和探索阶段流转
- 改动边界清晰，避免一次性改写全部主线 / 战斗 / 结算逻辑
- 能保留已有探索内容资产，只需将其组织方式从“动作列表”升级为“地图节点”

## 4. 玩家体验流程

### 4.1 进入地图

当主线推进到特定节点时，进入探索地图，而不是直接进入单个探索动作列表。

进入地图后：

- 页面显示地图标题、地点、当前探索目标
- 地图节点全部可见
- 玩家可以自行选择优先访问哪一类节点

### 4.2 地图内操作

玩家可以反复在节点间切换，执行以下行为：

- 在商铺购买物品
- 在武学点习得武学
- 在支线点完成一次固定事件
- 在情报点获取关键线索或物件

执行这些行为后：

- 奖励立即生效
- 写入章节日志
- 仍然停留在探索地图中
- 不会自动跳回主线

### 4.3 离开地图

地图始终存在一个离开节点，文案可统一为“收束行程”或“结束探索，返回主线”。

只有玩家主动点击该节点时，才会：

- 退出 exploration phase
- 切回挂接的主线节点
- 保留地图中的全部收益
- 在主线页展示一条简洁的探索回声

## 5. 数据模型设计

### 5.1 GameState 扩展

在 `GameState`（`src/game/types.ts:6`）中新增以下字段：

- `silver: number`：当前银两
- `currentExplorationMapId?: string`：当前停留的探索地图 ID
- `explorationReturnNodeId?: string`：本次探索结束后返回的主线节点 ID
- `completedMapNodes: string[]`：已完成的地图节点（例如 `mapId:nodeId`）
- `purchasedShopItems: string[]`：已购买商品（例如 `mapId:nodeId:itemId`）
- `completedSideQuests: string[]`：已完成支线（例如 `questId`）

这些状态用于支持：

- 节点的“已完成 / 已购 / 已习得”显示
- 首次有效逻辑
- 返回主线前持续停留在地图中

### 5.2 新增类型

在 `src/game/types.ts` 中新增以下类型：

#### `ExplorationMapDefinition`

- `id`
- `title`
- `location`
- `summary`
- `returnNodeId`
- `nodes: ExplorationMapNode[]`

#### `ExplorationMapNode`

- `id`
- `title`
- `description`
- `type: 'shop' | 'sidequest' | 'training' | 'intel' | 'exit'`
- `once?: boolean`
- `requires?: ChoiceRequirements`
- `shopItems?: ShopItemDefinition[]`
- `quest?: SideQuestDefinition`
- `reward?: ExplorationReward`
- `log?: string`

#### `ShopItemDefinition`

- `id`
- `name`
- `description`
- `price`
- `reward`
- `once?: boolean`

#### `SideQuestDefinition`

- `id`
- `title`
- `description`
- `log`
- `reward`
- `effects?`

说明：`training` 节点仍优先沿用已有 `ExplorationReward.martialSkillId`，不额外拆一套完全独立的武学奖励结构，以控制首版复杂度。

## 6. 节点类型设计

### 6.1 商铺节点

提供固定商品列表，使用银两购买。每个商品默认仅可购买一次。

首版商品类型：

- 恢复类：伤药、回气散
- 功能类：火折子、开锁针、情报纸签
- 普通兵刃：短刀、旧剑、铁尺等

购买成功后：

- 扣除银两
- 奖励加入库存 / 装备 / 日志
- 保持停留在地图

### 6.2 支线节点

首版采用“单节点固定小事件”模式，不做多阶段任务链。

奖励类型可以是：

- 银两
- 物件
- 声望 / 倾向数值
- 标记

节点完成后状态变为“已了结”。

### 6.3 武学节点

每个地图通常设置一个武学节点，产出一门固定武学。

示例映射：

- 酒楼横梁 → 燕行步
- 渡口浅滩 → 踏浪诀
- 盐栈廊桥 → 踏栈步

已习得时显示“已习得”，不重复发放。

### 6.4 情报节点

用于承接剧情前置信息和关键物件，例如：

- 封条残页
- 黑船名录
- 盐票碎片
- 口供副本

这些奖励可以直接服务主线中的 `requires.inventory` / `requires.flags` 条件，兼容现有主线判断系统（`src/game/types.ts:119`）。

### 6.5 离开节点

离开节点始终可见且始终存在。

它是地图内唯一允许改变“探索阶段 → 主线阶段”的节点；其他节点只会修改资源与状态，不改变主线推进。

## 7. 页面与交互设计

### 7.1 页面结构

现有 `ExplorationScreen`（`src/components/ExplorationScreen.tsx:47`）将从“单列表动作页”升级为“地图总览 + 节点详情”页面，但整体仍沿用当前探索页面的视觉风格。

页面分为四部分：

1. **顶部探索头区**
   - 地图名
   - 地点
   - 当前探索目标
   - 状态摘要（银两 / 已完成节点数 / 可返回主线提示）

2. **地图节点总览区**
   - 全部节点卡片直接可见
   - 每个节点展示名称、类型标签、简述和状态

3. **节点详情区**
   - 点击节点后展示详细内容和按钮
   - 商店展示商品列表
   - 支线展示事件描述和执行按钮
   - 武学展示武学介绍和习得按钮
   - 情报展示搜查说明和领取按钮
   - 离开展示“结束探索并返回主线”按钮

4. **探索所得 / 江湖回声区**
   - 节点执行后显示结果文案
   - 页面仍保留在探索地图中，不自动离开

### 7.2 交互原则

- 全部节点从一开始就可见
- 节点行为结果即时反馈
- 已完成节点要显式区分状态
- 离开动作必须清晰、可见、可重复理解
- 不通过复杂连线地图表达空间关系，首版用“地图节点卡片”表现地图语义

## 8. 引擎与状态流转

### 8.1 当前问题

当前探索引擎 `applyExplorationAction()` 会在结算时直接把 `currentNodeId` 切回主线节点，见 `src/game/engine/explorationEngine.ts:46`。这与“玩家手动结束探索再回主线”的需求冲突。

### 8.2 新的状态流转

#### 进入探索地图

主线触发探索时：

- phase 切为 `exploration`
- 设置 `currentExplorationMapId`
- 设置 `explorationReturnNodeId`
- 进入地图页面

#### 地图内执行节点

执行商店 / 支线 / 武学 / 情报节点时：

- 复制并更新 `GameState`
- 修改银两、背包、武学、flags、日志等
- 记录节点完成状态
- 不修改 `currentNodeId`
- 不退出 exploration phase

#### 结束探索

执行离开节点时：

- 读取 `explorationReturnNodeId`
- 设置 `currentNodeId = returnNodeId`
- 清空 `currentExplorationMapId`
- 清空 `explorationReturnNodeId`
- 退出 exploration，回到 story phase

## 10. 测试策略

### 10.1 引擎测试

新增测试覆盖：

1. 进入地图后不会自动回主线
2. 购买商品会扣除银两并加入物品
3. 武学节点只能习得一次
4. 支线节点完成后会记录状态
5. 点击离开节点后才会回主线

### 10.2 组件测试

新增 `ExplorationScreen` 相关测试覆盖：

1. 地图节点全部可见
2. 点击节点后详情区更新
3. 已完成 / 已购 / 已习得状态正确显示
4. 离开节点始终存在
5. 执行节点动作后页面仍停留在探索地图

### 10.3 App 集成测试

新增一条完整路径：

- 主线进入地图
- 购买一个物品
- 习得一门武学
- 完成一个支线
- 手动返回主线
- 主线页面能正确承接这些收益

## 11. 风险与边界

### 11.1 主要风险

- 现有探索结构是“动作即返回主线”，需要把“地图内行为”和“主线推进”拆开
- `ExplorationScreen` 页面将从简单动作列表升级为复合页面，需要谨慎控制视觉密度
- 奖励系统会从单纯武学 / 兵刃扩展到货币和商店库存，需要保证状态去重逻辑清晰

### 11.2 控制策略

- 首版只做固定内容，避免随机性带来的验证复杂度
- 首版只做三张地图，避免内容膨胀
- 复用现有 `ChoiceRequirements`、`ExplorationReward` 与日志机制，降低新模型数量

## 12. 实施原则

1. 优先保持现有主线与战斗系统不受扰动
2. 先建立地图层和引擎层，再补 UI
3. 先做一张图打通，再抽象到三张图复用
4. 所有“返回主线”的行为必须显式、可测试、可追踪
