# 武侠 Web MVP 设计说明

## 1. 项目目标

构建一个浏览器可运行的武侠剧情 RPG 原型，用最小范围验证以下 4 件事：

1. 剧情分支是否具备吸引力。
2. 状态变量是否能让选择产生明确后果感。
3. 轻量回合战斗是否能提供武侠感。
4. 玩家是否愿意为不同结果反复试玩。

该原型不是完整游戏产品，而是一个可玩、可验证、可扩展的 Web 单机版闭环。

## 2. 范围边界

### 本次实现包含

- 开始页。
- 角色创建页。
- 剧情推进页。
- 轻量回合制战斗页。
- 章节结算页。
- 结局页。
- 本地自动存档与重开。

### 本次实现不包含

- 大地图自由探索。
- 复杂门派系统。
- 伙伴系统。
- 装备/掉落深度体系。
- 后端服务与云存档。
- 内容编辑器。

## 3. 核心体验闭环

玩家体验流程固定为：

开始游戏 → 创建角色 → 进入剧情节点 → 做出选择 → 触发非战斗结果或战斗 → 章节结算 → 进入下一个节点或结局。

第一版主线闭环只覆盖以下节点：

- `N00` 破庙夜雨
- `N10` 镖银失踪案
- `N11` 酒楼听风
- `E01` 正道倾向结局
- `E02` 黑市倾向结局
- `E03` 中立悬念结局

## 4. 技术方案

采用纯前端单页应用实现：

- React + TypeScript
- Vite
- 本地状态管理（React state/context）
- localStorage 本地持久化

原因：

- 搭建快，适合空仓起步。
- 能快速验证玩法核心。
- 内容和逻辑易于继续扩展。

## 5. 页面与状态阶段

应用用单页状态切屏，不先引入复杂路由。游戏阶段包括：

- `start`：开始页
- `create`：角色创建页
- `story`：剧情页
- `battle`：战斗页
- `result`：章节结算页
- `ending`：结局页

这样可以保证状态同步简单、实现成本可控，并让体验保持连续叙事流。

## 6. 角色创建设计

### 出身

第一版提供 3 个出身：

- 破落书生：叙事收益偏强，战斗略弱。
- 江湖浪客：攻势与灵活度更高。
- 镖局子弟：生命与防守更稳。

### 天赋

第一版提供 3 个天赋：

- 稳心：初始气更高。
- 听风：更容易得到额外线索或名望反馈。
- 强骨：初始生命与势更高。

角色创建完成后立即生成初始玩家状态，并进入 `N00`。

## 7. 剧情节点设计

剧情系统采用节点配置驱动。每个节点包含：

- `id`
- `title`
- `location`
- `content`
- `choices`
- 可选的战斗触发信息
- 结算文本或结局标记

### 节点目标

- `N00`：建立氛围并引导第一次关键抉择。
- `N10`：引入残篇去向与价值冲突。
- `N11`：提供多种解决方式，并将选择导向结局。

每个选项都必须带来：

- 至少一个状态变量变化。
- 至少一个即时反馈文本。
- 明确的后续节点或战斗结果。

## 8. 状态系统设计

使用统一 `GameState` 管理玩家与世界状态。

### 基础字段

- `name`
- `origin`
- `talent`

### 战斗字段

- `maxHp`
- `hp`
- `maxQi`
- `qi`
- `maxPoise`
- `poise`
- `attack`
- `defense`

### 叙事字段

- `morality`
- `wantedLevel`
- `reputation`
- `debt`
- `favorSword`
- `favorBlackMarket`

### 流程字段

- `currentNodeId`
- `visitedNodes`
- `flags`
- `chapterLogs`

其中 `flags` 用于记录布尔型剧情状态，例如是否救下密探、是否私藏残篇、是否赢下比武。

## 9. 战斗设计

### 战斗目标

战斗必须：

- 规则简单。
- 1 分钟内可以理解。
- 与剧情结果紧密绑定。
- 能体现武侠中的“调息、压势、破招”。

### 战斗资源

第一版仅保留 3 个核心资源：

- `HP`：生命值。
- `Qi`：内力资源。
- `Poise`：架势值。

当敌人 `Poise` 被压低时进入破绽状态，玩家可用 `破招` 获得更高收益。

### 玩家行动

- 普通攻击：稳定输出，轻度削势。
- 运气调息：回复气并小幅回复势。
- 防守架招：减伤并尝试反制强攻。
- 破招：消耗气，对破绽敌人造成高收益。

### 战斗结果

- 胜利：获得完整收益。
- 失败：进入负伤脱身结果，保留剧情推进，但承担代价。

## 10. 剧情与战斗联动

剧情选择直接影响战斗上下文：

- 出身影响基础属性。
- 某些选项影响是否先手、敌人势值或战斗奖励。
- 某些选项可完全避开战斗，但增加其他叙事代价。

因此战斗是剧情结果的一部分，而不是独立小游戏。

## 11. 本地存档

第一版使用 localStorage 做自动存档，触发时机：

- 角色创建完成后。
- 每次章节结算后。
- 进入结局后。

开始页提供：

- 继续游戏
- 重新开始

## 12. UI 风格

采用轻量古风界面：

- 深色背景
- 米白正文
- 金色/朱红强调色
- 轻微纸张/墨色纹理

目标是用排版与配色塑造武侠氛围，而非重资产美术。

## 13. 最低验收标准

### 功能链路

1. 能完成开始 → 创建角色 → `N00` → `N10` → `N11` → 结局的完整流程。
2. 至少有一条路径会进入战斗，且战斗可正常完成 3 回合以上。
3. 战斗失败不会卡死，而会进入失败后分支或结算。

### 状态正确性

1. 选项效果可正确修改变量。
2. 节点条件判断可驱动不同结果。
3. 战斗资源不会出现负值或异常越界。
4. 自动存档、刷新恢复、重开清档都正确。

### UI 可用性

1. 小屏宽度下页面不炸布局。
2. 长文本可滚动。
3. 战斗日志和状态栏实时更新。

## 14. 后续扩展方向

若 MVP 结果正向，下一步可扩展：

- 更多章节节点（如 `N12`、`N20`）。
- 更完整的门派和好感系统。
- 同行侠客与队伍协作。
- 更丰富的敌人配置与心法差异。
- 更强的内容配置能力。

## 15. 设计结论

本项目应优先做成一个“章节节点 + 状态变量 + 轻量回合战斗 + 多结局”的浏览器可玩原型，用最小但完整的实现验证武侠剧情 RPG 的核心吸引力。
