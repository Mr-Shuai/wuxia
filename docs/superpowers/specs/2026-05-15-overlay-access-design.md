# 游戏内悬浮入口与整页随览弹窗设计

## 1. 目标

为当前武侠 Web 原型增加一个统一的游戏内入口，用于查看：

- 背包
- 状态
- 剧情任务

该入口需要满足以下体验目标：

1. 不打断当前叙事或战斗页面的主流程。
2. 在所有游戏内页面都能以一致方式访问。
3. 默认优先展示“任务”，帮助玩家理解当前目标。
4. 用整页覆盖弹窗承载信息，而不是把过多内容挤进侧栏。

## 2. 已确认约束

本次设计已经确认以下交互约束：

- 入口形式：右下角单按钮。
- 入口范围：所有游戏内页面都显示。
- 打开方式：点击按钮后打开整页覆盖弹窗。
- 默认页签：任务。
- 可切换页签：任务 / 状态 / 背包。

## 3. 范围边界

### 本次实现包含

- 在 `story`、`exploration`、`battle`、`ending` 阶段显示统一入口。
- 整页覆盖弹窗容器。
- 任务 / 状态 / 背包三个页签及切换逻辑。
- 基于现有 `GameState` 和当前 `phase` 生成展示内容。
- 关闭行为：关闭按钮、遮罩点击、`Esc`。

### 本次实现不包含

- 完整任务系统或任务树。
- 装备管理、使用物品、丢弃物品等交互。
- 背包分类筛选、排序、批量整理。
- 多层弹窗或复杂二级详情页。
- 非游戏内页面（开始页、创建页）的入口展示。

## 4. 方案选择

### 4.1 入口方案

放弃“右侧多按钮快捷入口”，采用“右下角单按钮进入”。

原因：

- 当前页面已经在压缩主内容，多个悬浮按钮更容易分散注意力。
- 单按钮更像游戏中的“菜单 / 行囊 / 江湖册”入口，语义更统一。
- 后续如果增加未读提示、任务更新提示，也可以挂在单按钮上而不破坏布局。

### 4.2 弹窗方案

放弃侧边栏切换和小浮层方案，采用整页覆盖弹窗。

原因：

- 当前主线页、探索页、战斗页都在往首屏完整展示方向收缩，继续往侧栏堆信息会破坏这个目标。
- 状态、背包、任务三类信息结构不同，整页弹窗更适合承载分页内容。
- 覆盖弹窗可以保持当前 phase 不变，不影响游戏流程状态。

## 5. 交互设计

### 5.1 入口按钮

右下角固定悬浮一个单按钮。

建议视觉语义：

- 文案优先使用“江湖册”或“随览”。
- 按钮尺寸偏小，保持常驻但不抢主视觉。
- 打开弹窗后，按钮可以保持激活态，或者在弹窗打开时隐藏均可；优先采用隐藏，避免视觉重叠。

### 5.2 打开行为

- 点击入口按钮，打开整页覆盖弹窗。
- 首次打开和每次重新打开时，默认页签都是“任务”。

### 5.3 关闭行为

- 点击右上角关闭按钮。
- 点击遮罩层。
- 键盘 `Esc`。

### 5.4 页签切换

弹窗顶部提供 3 个页签：

- 任务
- 状态
- 背包

切换规则：

- 切换只改变弹窗内容，不改变游戏 phase。
- 切换不清空当前页面的战斗、探索、剧情状态。
- 关闭后再次打开，始终回到任务页签，不记住上次停留页签。

## 6. 信息架构

### 6.1 任务页签（默认）

任务页签承担“我现在在做什么”的核心说明。

展示内容：

1. 当前阶段：主线 / 探索 / 战斗 / 结局余波。
2. 当前节点标题。
3. 当前地点。
4. 当前目标一句话说明。
5. 最近关键日志 2~3 条。

任务页签不是完整任务系统，而是“当前目标总览”。

文案生成规则：

- `story`：围绕当前剧情节点给出“继续推进主线”的目标描述。
- `exploration`：围绕当前探索场景给出“摸清线索并返回主线”的目标描述。
- `battle`：围绕当前战斗给出“赢下当前交锋”或“稳住局势”的目标描述。
- `ending`：围绕当前结局给出“查看余波并决定是否继续前行”的目标描述。

### 6.2 状态页签

状态页签承担“完整人物状态总览”。

展示内容：

- 核心资源：生命 / 气 / 势。
- 江湖阅历：侠义 / 通缉 / 名望 / 人情债。
- 当前兵刃。
- 已学武学。

相比当前页面侧栏中的紧凑展示，这里使用更完整的卡片分组，不再压缩信息层级。

### 6.3 背包页签

背包页签承担“当前持有内容查看”。

展示内容：

- 当前装备兵刃。
- 背包物件列表。
- 已学武学列表。
- 空态提示。

本次不提供装备切换、使用、丢弃等操作，仅做只读查看。

## 7. 数据来源设计

### 7.1 顶层状态来源

入口显隐和弹窗状态统一由 `App` 顶层管理，当前页面组件不各自持有弹窗开关状态。

已有集中入口位置：`src/App.tsx:106`。

新增顶层 UI 状态：

- `isOverlayOpen`
- `activeOverlayTab`

### 7.2 任务页签数据来源

直接从现有数据派生：

- `phase`
- `gameState.currentNodeId`
- `gameState.visitedNodes`
- `gameState.chapterLogs`

必要时复用：

- 主线节点配置 `storyNodes`
- 探索场景配置 `explorationScenes`
- `explorationEntryByNode` 映射：`src/App.tsx:23`

任务文案采用轻规则映射，不单独引入任务状态树。

### 7.3 状态页签数据来源

直接复用当前状态侧栏已经使用的数据组织方式：

- 当前兵刃与武学：`src/components/StatusPanel.tsx:19`
- 核心资源：`src/components/StatusPanel.tsx:23`
- 阅历数据：`src/components/StatusPanel.tsx:28`

实现时不直接复用 `StatusPanel` 本体，以避免侧栏布局样式和弹窗样式耦合。

### 7.4 背包页签数据来源

直接基于当前 `GameState`：

- `activeWeaponId`
- `inventory`
- `martialSkills`

如后续引入任务物件分类，可在该面板内部扩展分组规则。

## 8. 组件与结构设计

推荐采用“App 顶层统一控制 + 独立弹窗组件”的实现方式。

### 8.1 顶层控制

`src/App.tsx` 负责：

- 根据 `phase` 判断是否显示入口。
- 控制弹窗开关。
- 控制当前页签。
- 将 `phase`、`gameState` 传给弹窗内容面板。

### 8.2 新组件建议

- `GameQuickAccessButton`
  - 右下角悬浮单按钮。
- `GameOverlayPanel`
  - 整页覆盖弹窗容器。
- `TaskOverviewPanel`
  - 任务页签内容。
- `StatusOverviewPanel`
  - 状态页签内容。
- `InventoryOverviewPanel`
  - 背包页签内容。

### 8.3 页面边界

以下页面继续只负责正文：

- `src/components/StoryScreen.tsx:21`
- `src/components/ExplorationScreen.tsx:47`
- `src/components/BattleScreen.tsx`
- `src/components/EndingScreen.tsx`

不在这些页面内部重复实现入口按钮和弹窗逻辑。

## 9. 样式与视觉原则

### 9.1 入口按钮

- 固定在右下角。
- 使用与现有古风 UI 相容的深色半透明底。
- 保持单按钮、轻存在感。
- 支持 hover / active 状态。

### 9.2 整页弹窗

- 整体使用较深的遮罩背景，保持与现有古风主题统一。
- 弹窗主体居中，宽度大于当前侧栏，但不占满浏览器边缘。
- 小屏时允许纵向滚动。

### 9.3 内容结构

- 任务、状态、背包都使用卡片化分组。
- 保证“当前目标”信息优先级最高。
- 不复用当前页面右侧紧凑侧栏的压缩视觉层级。

## 10. 错误处理与边界情况

### 10.1 无 `gameState`

当不处于游戏内 phase 或 `gameState` 缺失时：

- 不展示入口按钮。
- 不允许弹窗打开。

### 10.2 数据为空

- 背包为空时展示空态文案。
- 武学为空时展示“暂无已学武学”。
- 日志不足 2~3 条时按实际数量显示，不补占位。

### 10.3 战斗中打开弹窗

- 允许打开。
- 不暂停业务状态推进，只冻结用户当前视图交互焦点。
- 关闭弹窗后回到原战斗视图。

### 10.4 结局页打开弹窗

- 允许打开。
- 任务页签显示当前结局标题与继续/重开相关的当前语境。

## 11. 测试策略

至少覆盖以下验证：

1. `start` / `create` 不显示入口。
2. `story` / `exploration` / `battle` / `ending` 显示入口。
3. 点击单按钮后弹窗打开。
4. 默认页签为任务。
5. 可以切换到状态和背包。
6. 关闭按钮、遮罩、`Esc` 都能关闭。
7. 任务页签能根据不同 `phase` 显示对应内容。
8. 状态页签能显示核心资源与江湖阅历。
9. 背包页签能显示装备、物品与武学。

## 12. 分步实施建议

建议实施顺序：

1. 在 `App` 顶层加入入口状态与弹窗开关。
2. 加入右下角单按钮，并限制显示范围。
3. 加入整页弹窗容器与关闭交互。
4. 先完成任务页签。
5. 再接入状态页签。
6. 最后接入背包页签并补空态。
7. 完成交互测试与样式回归。

## 13. 成功标准

本次设计实现完成后，应满足：

1. 玩家在所有游戏内页面都能通过右下角单按钮进入总览弹窗。
2. 打开后默认看到任务，不需要先理解复杂菜单。
3. 状态与背包信息从页面侧栏中解耦出来，方便后续继续压缩正文首屏。
4. 不引入新的全局状态混乱，也不让各页面分别维护一套入口逻辑。
